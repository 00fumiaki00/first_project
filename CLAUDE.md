## 今日学んだ概念（Day 1）

### プログラムの基本
- プログラム＝コンピュータへの命令書
- 3つの基本構造：順番に実行・条件で分岐・繰り返す
- コンピュータは0と1しか理解できない
- PythonはAIの道具が一番揃っている言語

### ツールの全体像
```
VS Code（作業場）
└── ターミナル（内蔵）
└── Claude Code（内蔵・あなた）

Python（言語）
└── ライブラリ（既製品の部品）
└── フレームワーク（土台ごと借りる）

API（外から借りるインフラ）
データベース（アプリの記憶）
GitHub（コードの倉庫・履歴書）
```

### 重要概念
- **変数**＝データに貼るラベル（箱ではなく名札）
- **関数**＝繰り返す作業を一回定義して呼び出す仕組み
- **クラス**＝材料と動作がセットになった設計図
- **インスタンス**＝設計図から作った実物
- **オブジェクト指向**＝現実世界の構造をそのままプログラムに持ち込む思想
- **データ構造**＝リスト・辞書・タプル・セットの4種類
- **非同期処理**＝待ち時間を無駄にしない仕組み（async/await）
- **API**＝誰かが作った関数を外部から呼び出す仕組み
- **ライブラリ**＝既製品の部品（pip installで入れる）
- **フレームワーク**＝土台ごと借りる仕組み
- **データベース**＝アプリの記憶（SQL/NoSQL）
- **クラウド**＝24時間動かし続ける場所
- **GitHub**＝コードの倉庫・デプロイの起点
- **デプロイ**＝倉庫から取り出して実際に動かすこと
- **認証**＝お前は誰かを確認する仕組み
- **セキュリティ**＝APIキーを隠す・パスワードを暗号化する

### 設計の思想
- 設計＝作るものを小さな部品に分解すること
- 全てのプログラムは「入力と出力」で説明できる
- シンプルに・シンプルに・説明できる状態にする

### Gitの概念
```
git status　→　変更を確認する
git add　　→　ステージに上げる
git commit　→　写真を撮る（変更を記録する）
git push　　→　GitHubに送る
```
- ブランチ＝試作する場所・本番を壊さない
- コミットメッセージは「何をなぜ変えたか」を書く

### コードの作法
1. 名前は意味のある言葉にする
2. コメントは「なぜ」を書く
3. 一つの関数は一つのことだけやる
4. 同じコードを繰り返さない
5. インデントを揃える

**まず動くコードを書く→動いたら整える**

### Day 4で習得したこと
- エスケープの概念（`\n`＝改行の合図、`\"`＝ダブルクォート文字そのものの合図）
- プロンプト内で「[Aor B]」のように書くことで、AIに条件分岐を自然言語のまま判断させる書き方
- コードとデータの分離の実感：パターン2・3を追加する間、mail_generator.py（ロジック）は一度も編集せず、patterns.json（データ）の追加だけで機能拡張できた
- Version 1とVersion 2の役割の違い：V2（管理画面）は入力の技術的な手間（JSON構文）を減らすものであり、業務ロジックを考える作業自体を代替するものではない
- 効率化・自動化を志向する姿勢はプログラマーの美徳（怠惰・短気・傲慢）の一つであり、正しいタイミングで発揮すべきという考え方
- パターン判定バグのデバッグ経験：keywordsを増やしても直らなかった原因は「判定ロジック（detection_prompt）に優先順位のルールが無かったこと」。表面的な対処（keyword追加）で直らない時は、ロジック側の設計を疑うという原因追及の手順を体験した
- 判定基準は「どの単語が含まれるか」ではなく「指示書の結末（最後の動作）に一致するパターンを優先する」という明示的なルールをdetection_promptに追加して解決
- 「配達を伴うかどうか」のような業務ルールは、個別パターンのprompt内に書いても判定ロジック（detection_prompt）には見えない。判定に使いたい情報は判定用プロンプトの方に明示する必要がある
- 存在しないケース（配達を伴わない単独のシャーシ切り）に対応したパターンを作っていたことに気づき、削除して別パターンに統合。keywordsの衝突は「本当にそのパターンが独立して必要か」を疑うきっかけにもなる
- AIの回答は指示通り「数字だけ」にならないことがある（余計な説明文が付き、max_tokensで途中に千切れてStopIterationが発生）。`.splitlines()[0]`で最初の1行だけを取り出す形にして安定化
- 1回のプロンプトに複数の長いテンプレートを詰め込んで条件分岐させると、AIが文章量の多い方に引っ張られて誤答しやすい（未解決の課題として次回に持ち越し）

### Day 5で習得したこと
- 「答えが決定的に決まる判断」をAIに毎回判断させると不安定になる。今回は「配達先が千歳/札幌/道東ベースかどうか」という固定キーワードの一致チェックをAIに判断させていたのが不安定の根本原因だった。これをPythonの`if`/`in`/`or`だけで判定するよう変更し、AIには「分岐が決まった後の、テンプレートへの当てはめ」だけをやらせる設計にしたら安定した
- 判断をコードに移した後、patterns.jsonのpattern4を`prompt_output1`/`prompt_output2`/`prompt_output3`の3つに分割。「出力1＋2」と「出力1＋3」でテンプレートを丸ごと2つ複製する案（案A）と、3つに分割してPython側で組み合わせる案（案B）を比較し、保守性（修正箇所が少ない方が良い）を理由に案Bを選んだ
- 分割した3つのプロンプトをただ`+`で連結すると、それぞれが持っていた「以下の指示書から情報を読み取り、〜生成してください」という指示文が重複し、AIが混乱して2通目の内容が欠落するバグが発生した。指示文は1つにまとめ、mail_generator.py側で分岐ごとに動的に組み立てる形に修正して解決
- AIは「テンプレート通り忠実に再現する」のではなく「たぶん必要だろうもの」を予測して生成する、という性質を実際のバグで体験した。省略されると困る空欄フィールド群をAIが勝手に省略していた。個別に項目名を列挙して「空欄で出力すること」と指示する方法は書き漏れ・タイプミスのリスクがあるため、「〇〇から〇〇までの項目は省略せず出力すること」という範囲指定の書き方に変更して解決
- `message.stop_reason`を見ることで、「max_tokensで強制的に打ち切られた」のか「AI自身が意図的にそこで終えた（end_turn）」のかを区別できることを学んだ。原因の切り分けに使える
- Pythonのエラー文（トレースバック）は基本的に一番下の1行だけ読めばいい。上に並ぶ`File "..." line ... in ...`は経路情報で、デバッグの初手ではほぼ無視してよい。最後の1行に「エラーの種類：内容」の形で本質が書いてある
- テンプレートの正しい項目名は「辞書的に正しいかどうか」ではなく「実際に会社で使われている名称に合わせる」のが正解。現場の実情を知ってる本人の判断が最優先される

### Day 6で習得したこと
- `pip install`はプロジェクトではなく「今使っているPythonの実行環境」にライブラリを入れる操作。別PCの別Python環境には反映されないため、GitHubにコードを上げても、そのPCで使うライブラリ自体は別途インストールし直す必要がある
- `requirements.txt`：プロジェクトが依存するライブラリを1ファイルにまとめておく仕組み。`pip install -r requirements.txt`で一括インストールでき、環境構築の再現性を確保できる
- `.env`も同じ理由で各PCに個別に作り直す必要がある（`.gitignore`で除外されているため、GitHub経由では運ばれない）。秘匿性は`.env`というファイル自体にあるのではなく、「GitHubという共有場所に届く経路を`.gitignore`で断っていること」にある
- `python-dotenv`の`load_dotenv()`が`.env`を読んで環境変数に登録し、`anthropic`ライブラリ側が`ANTHROPIC_API_KEY`という決まった名前の環境変数を自動で探しに行く、という2段階の仕組みで動いている
- 文字列（`"""..."""`）の中に`#`を書いてもPythonのコメントアウトにはならない。コメントアウトはコードの世界だけのルールで、文字列の中はただの文字データとして扱われる
- すでにgitでコミット済みの変更なら、削除して検証しても`git restore`（＝`git checkout --`の現代的な書き方）で直前のコミット状態に戻せる。gitに記録されていないものだけ、手動の退避が必要になる
- Day5で追加したdetection_promptの優先順位ルールが、無関係なパターン1/2/3の判定まで誤爆させていた（詳細はTROUBLESHOOTING.md参照）。ルールを名指し組（パターン名を明示）と曖昧組（「関連する方」のようにぼかした表現）に分けて検証し、曖昧な表現が誤爆の原因だと特定した
- 実証実験の結果、ルールを丸ごと削除しても全パターンが正しく判定されることを確認。YAGNI原則（今必要じゃないものは足さない）に従い、名指し組のルールも含めて復活させない判断をした。人間のチェックが入る下書き生成ツールでは「先回りの保険」より「壊れたら直す」の方が安全な場合がある
- コードとAIの使い分けの基準：条件を全部書き出せる判断（YES/NO、キーワード一致など）はコード（if文）、書き方の幅が無限で列挙できない自然文の理解・生成はAI、という住み分けが本質

### Day 7で習得したこと（途中）
- ファイルへの「追記」（`open(..., "a")`）と、Pythonのリストの`.append()`メソッドは全くの別物。JSONファイルは全体で1つの完成した構造なので、末尾にテキストを追記すると構文が壊れる。正しくは`json.load()`で全体をPythonのデータに変換→リストに`.append()`で追加→`json.dump()`で全体を書き直す、という3段階が必要
- 実装方法に複数の選択肢がある時、「簡単な方が難しい方の前提知識になっているか（依存関係があるか）」を見極める基準を得た。前提知識でないなら、簡単な方を経由せず実務レベルのやり方に直行してよい（YAGNIとは別の判断軸）
- 複数行の長いテキストを対話的に入力させる方法として、番兵方式（終了の合図となる文字列が来るまで1行ずつ読み込む）と、外部エディタ方式（`git commit`と同じ仕組み。tempfileで一時ファイルを作りnotepadを起動、閉じたら中身を読み込む）を比較。後者が実務でよく使われる方式と判明
- 「常に含む部分（common）」と「状況によって1つ選ぶ部分（branches）」という構造は、パターン4だけの特殊事情ではなく、パターン3（クレート配達の有無で出力2の内容が変わる）にも実際に必要なことが実指示書から判明。これにより`if pattern_id == "4"`という特定パターンへのハードコードをやめ、JSON側にデータとして持たせる汎用設計（案A）を選択した
- 分岐の「デフォルト（何にも一致しなかった時の受け皿）」を空リストで暗黙的に表現する案は、意味が読み取りにくい・並び順に依存するというリスクがあるため却下し、`"default": true`という明示的なフラグを採用（Explicit is better than implicit）
- パターン1についても、集荷が複数回（現場確認で1〜2回、3回以上はない）ある指示書に対応する必要があると判明。これは分岐ではなく「工程の数に応じて生成通数が変わる」別種の課題

## 重要な原則

> コードが書ける・意味がわかることに全ては集約されて繋がる。
> AIがどれだけ進化しても残るのは「判断できる人間」だ。
> 今日手に入れたのは地図だ。歩くのはFinだ。
> 継続するかどうかだけが唯一の変数だ。

---

## 学習法：フェインマン・テクノロジー

**「自分の言葉で説明できない概念は理解していない」**

### Claudeとの学習サイクル
```
Claudeが質問する
　↓
Finが直感で答える
　↓
答え合わせ・補足
　↓
自分の言葉でもう一度説明する
　↓
理解の確認完了
```

### ヒントの段階的な外し方
```
完全初心者の今　→　ヒントありで考える習慣を作る
慣れてきたら　　→　ヒントを減らしていく
実務レベル　　　→　完全自力で設計・実装できる
```

**最終的に自分の言葉で説明できるかどうかが理解の判断基準だ。**

## 普遍的な思考回路：分解と統合

**「難しいことを細かく分解していくことで分かりやすくなる」**

```
コードの設計　→　大きな問題を小さな部品に分解して組み合わせる
学習の設計　　→　難しい概念を小さな問いに分解して繋げる
人生の設計　　→　大きなビジョンを小さなステップに分解して積み上げる
```

**全部同じ構造だ。この思考回路はどの領域でも機能する普遍的な武器だ。**
