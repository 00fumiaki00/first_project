## Day 1（2026/06/27）

### プログラムの基本
- プログラム＝コンピュータへの命令書
- 3つの基本構造：順番に実行・条件で分岐・繰り返す
- コンピュータは0と1しか理解できない
- PythonはAIの道具が一番揃っている言語

### ツールの全体像
```
VS Code（作業場）
└── ターミナル（内蔵）
└── Claude Code（内蔵・あなた）

Python（言語）
└── ライブラリ（既製品の部品）
└── フレームワーク（土台ごと借りる）

API（外から借りるインフラ）
データベース（アプリの記憶）
GitHub（コードの倉庫・履歴書）
```

### 重要概念
- **変数**＝データに貼るラベル（箱ではなく名札）
- **関数**＝繰り返す作業を一回定義して呼び出す仕組み
- **クラス**＝材料と動作がセットになった設計図
- **インスタンス**＝設計図から作った実物
- **オブジェクト指向**＝現実世界の構造をそのままプログラムに持ち込む思想
- **データ構造**＝リスト・辞書・タプル・セットの4種類
- **非同期処理**＝待ち時間を無駄にしない仕組み（async/await）
- **API**＝誰かが作った関数を外部から呼び出す仕組み
- **ライブラリ**＝既製品の部品（pip installで入れる）
- **フレームワーク**＝土台ごと借りる仕組み
- **データベース**＝アプリの記憶（SQL/NoSQL）
- **クラウド**＝24時間動かし続ける場所
- **GitHub**＝コードの倉庫・デプロイの起点
- **デプロイ**＝倉庫から取り出して実際に動かすこと
- **認証**＝お前は誰かを確認する仕組み
- **セキュリティ**＝APIキーを隠す・パスワードを暗号化する

### 設計の思想
- 設計＝作るものを小さな部品に分解すること
- 全てのプログラムは「入力と出力」で説明できる
- シンプルに・シンプルに・説明できる状態にする

### Gitの概念
```
git status　→　変更を確認する
git add　　→　ステージに上げる
git commit　→　写真を撮る（変更を記録する）
git push　　→　GitHubに送る
```
- ブランチ＝試作する場所・本番を壊さない
- コミットメッセージは「何をなぜ変えたか」を書く

### コードの作法
1. 名前は意味のある言葉にする
2. コメントは「なぜ」を書く
3. 一つの関数は一つのことだけやる
4. 同じコードを繰り返さない
5. インデントを揃える

**まず動くコードを書く→動いたら整える**

## Day 4で習得したこと
- エスケープの概念（`\n`＝改行の合図、`\"`＝ダブルクォート文字そのものの合図）
- プロンプト内で「[Aor B]」のように書くことで、AIに条件分岐を自然言語のまま判断させる書き方
- コードとデータの分離の実感：パターン2・3を追加する間、mail_generator.py（ロジック）は一度も編集せず、patterns.json（データ）の追加だけで機能拡張できた
- Version 1とVersion 2の役割の違い：V2（管理画面）は入力の技術的な手間（JSON構文）を減らすものであり、業務ロジックを考える作業自体を代替するものではない
- 効率化・自動化を志向する姿勢はプログラマーの美徳（怠惰・短気・傲慢）の一つであり、正しいタイミングで発揮すべきという考え方
- パターン判定バグのデバッグ経験：keywordsを増やしても直らなかった原因は「判定ロジック（detection_prompt）に優先順位のルールが無かったこと」。表面的な対処（keyword追加）で直らない時は、ロジック側の設計を疑うという原因追及の手順を体験した
- 判定基準は「どの単語が含まれるか」ではなく「指示書の結末（最後の動作）に一致するパターンを優先する」という明示的なルールをdetection_promptに追加して解決
- 「配達を伴うかどうか」のような業務ルールは、個別パターンのprompt内に書いても判定ロジック（detection_prompt）には見えない。判定に使いたい情報は判定用プロンプトの方に明示する必要がある
- 存在しないケース（配達を伴わない単独のシャーシ切り）に対応したパターンを作っていたことに気づき、削除して別パターンに統合。keywordsの衝突は「本当にそのパターンが独立して必要か」を疑うきっかけにもなる
- AIの回答は指示通り「数字だけ」にならないことがある（余計な説明文が付き、max_tokensで途中に千切れてStopIterationが発生）。`.splitlines()[0]`で最初の1行だけを取り出す形にして安定化
- 1回のプロンプトに複数の長いテンプレートを詰め込んで条件分岐させると、AIが文章量の多い方に引っ張られて誤答しやすい（未解決の課題として次回に持ち越し）

## Day 5で習得したこと
- 「答えが決定的に決まる判断」をAIに毎回判断させると不安定になる。今回は「配達先が千歳/札幌/道東ベースかどうか」という固定キーワードの一致チェックをAIに判断させていたのが不安定の根本原因だった。これをPythonの`if`/`in`/`or`だけで判定するよう変更し、AIには「分岐が決まった後の、テンプレートへの当てはめ」だけをやらせる設計にしたら安定した
- 判断をコードに移した後、patterns.jsonのpattern4を`prompt_output1`/`prompt_output2`/`prompt_output3`の3つに分割。「出力1＋2」と「出力1＋3」でテンプレートを丸ごと2つ複製する案（案A）と、3つに分割してPython側で組み合わせる案（案B）を比較し、保守性（修正箇所が少ない方が良い）を理由に案Bを選んだ
- 分割した3つのプロンプトをただ`+`で連結すると、それぞれが持っていた「以下の指示書から情報を読み取り、〜生成してください」という指示文が重複し、AIが混乱して2通目の内容が欠落するバグが発生した。指示文は1つにまとめ、mail_generator.py側で分岐ごとに動的に組み立てる形に修正して解決
- AIは「テンプレート通り忠実に再現する」のではなく「たぶん必要だろうもの」を予測して生成する、という性質を実際のバグで体験した。省略されると困る空欄フィールド群をAIが勝手に省略していた。個別に項目名を列挙して「空欄で出力すること」と指示する方法は書き漏れ・タイプミスのリスクがあるため、「〇〇から〇〇までの項目は省略せず出力すること」という範囲指定の書き方に変更して解決
- `message.stop_reason`を見ることで、「max_tokensで強制的に打ち切られた」のか「AI自身が意図的にそこで終えた（end_turn）」のかを区別できることを学んだ。原因の切り分けに使える
- Pythonのエラー文（トレースバック）は基本的に一番下の1行だけ読めばいい。上に並ぶ`File "..." line ... in ...`は経路情報で、デバッグの初手ではほぼ無視してよい。最後の1行に「エラーの種類：内容」の形で本質が書いてある
- テンプレートの正しい項目名は「辞書的に正しいかどうか」ではなく「実際に会社で使われている名称に合わせる」のが正解。現場の実情を知ってる本人の判断が最優先される

## Day 6で習得したこと
- `pip install`はプロジェクトではなく「今使っているPythonの実行環境」にライブラリを入れる操作。別PCの別Python環境には反映されないため、GitHubにコードを上げても、そのPCで使うライブラリ自体は別途インストールし直す必要がある
- `requirements.txt`：プロジェクトが依存するライブラリを1ファイルにまとめておく仕組み。`pip install -r requirements.txt`で一括インストールでき、環境構築の再現性を確保できる
- `.env`も同じ理由で各PCに個別に作り直す必要がある（`.gitignore`で除外されているため、GitHub経由では運ばれない）。秘匿性は`.env`というファイル自体にあるのではなく、「GitHubという共有場所に届く経路を`.gitignore`で断っていること」にある
- `python-dotenv`の`load_dotenv()`が`.env`を読んで環境変数に登録し、`anthropic`ライブラリ側が`ANTHROPIC_API_KEY`という決まった名前の環境変数を自動で探しに行く、という2段階の仕組みで動いている
- 文字列（`"""..."""`）の中に`#`を書いてもPythonのコメントアウトにはならない。コメントアウトはコードの世界だけのルールで、文字列の中はただの文字データとして扱われる
- すでにgitでコミット済みの変更なら、削除して検証しても`git restore`（＝`git checkout --`の現代的な書き方）で直前のコミット状態に戻せる。gitに記録されていないものだけ、手動の退避が必要になる
- Day5で追加したdetection_promptの優先順位ルールが、無関係なパターン1/2/3の判定まで誤爆させていた。ルールを名指し組（パターン名を明示）と曖昧組（「関連する方」のようにぼかした表現）に分けて検証し、曖昧な表現が誤爆の原因だと特定した
- 実証実験の結果、ルールを丸ごと削除しても全パターンが正しく判定されることを確認。YAGNI原則（今必要じゃないものは足さない）に従い、名指し組のルールも含めて復活させない判断をした。人間のチェックが入る下書き生成ツールでは「先回りの保険」より「壊れたら直す」の方が安全な場合がある
- コードとAIの使い分けの基準：条件を全部書き出せる判断（YES/NO、キーワード一致など）はコード（if文）、書き方の幅が無限で列挙できない自然文の理解・生成はAI、という住み分けが本質

## Day 7で習得したこと
- ファイルへの「追記」（`open(..., "a")`）と、Pythonのリストの`.append()`メソッドは全くの別物。JSONファイルは全体で1つの完成した構造なので、末尾にテキストを追記すると構文が壊れる。正しくは`json.load()`で全体をPythonのデータに変換→リストに`.append()`で追加→`json.dump()`で全体を書き直す、という3段階が必要
- 実装方法に複数の選択肢がある時、「簡単な方が難しい方の前提知識になっているか（依存関係があるか）」を見極める基準を得た。前提知識でないなら、簡単な方を経由せず実務レベルのやり方に直行してよい（YAGNIとは別の判断軸）
- 複数行の長いテキストを対話的に入力させる方法として、番兵方式（終了の合図となる文字列が来るまで1行ずつ読み込む）と、外部エディタ方式（`git commit`と同じ仕組み。tempfileで一時ファイルを作りnotepadを起動、閉じたら中身を読み込む）を比較。後者が実務でよく使われる方式と判明
- 「常に含む部分（common）」と「状況によって1つ選ぶ部分（branches）」という構造は、パターン4だけの特殊事情ではなく、パターン3（クレート配達の有無で出力2の内容が変わる）にも実際に必要なことが実指示書から判明。これにより`if pattern_id == "4"`という特定パターンへのハードコードをやめ、JSON側にデータとして持たせる汎用設計（案A）を選択した
- 分岐の「デフォルト（何にも一致しなかった時の受け皿）」を空リストで暗黙的に表現する案は、意味が読み取りにくい・並び順に依存するというリスクがあるため却下し、`"default": true`という明示的なフラグを採用（Explicit is better than implicit）
- パターン1についても、集荷が複数回（現場確認で1〜2回、3回以上はない）ある指示書に対応する必要があると判明。これは分岐ではなく「工程の数に応じて生成通数が変わる」別種の課題
- 辞書の`.get(キー, デフォルト値)`：キーが存在しない場合でもKeyErrorにならず代わりの値を返せる。branchesの中でdefault用のbranchだけ`"conditions"`キーを持たないため、`branch["conditions"]`だと確実に落ちる箇所を`branch.get("conditions", [])`に変えて回避した
- `any(条件 for 変数 in リスト)`：リストの中身のどれか1つでも条件を満たすかを、可変長のリストに対して汎用的にチェックできる書き方。今まで`or`でキーワードを1つずつ手動でつないでいた固定判定（three_base）を、この書き方に置き換えて汎用化した
- `break`：forループの中で使うと、その時点で残りの繰り返しを打ち切れる。一致するbranchが1つ見つかった後、無駄に残りをチェックし続けない意図を明確にできた
- 変数は代入された瞬間に初めて存在する。一度も代入されないまま参照すると`NameError`になる。forループが1つも一致せず終わった場合、`prompt`が代入されないまま後続処理に進んでしまう危険があった
- if/elseは自分に直接ぶら下がっている条件だけを見ており、中にネストしたfor/ifの結果を暗黙に引き継いだりしない。「一致しなければ外側のelseが実行されるはず」という誤解を実際に指摘されて修正した
- 「マッチしたかどうかを覚えておく目印用の変数（フラグ）」という設計パターン：`matched = False`で開始し、一致した時点で`True`に書き換え、forループを抜けた後`if not matched:`でdefault分岐を拾う、という一連の流れを自力で組み立てて実装した
- 実API込みでパターン3のクレート配達branchをテストし、①（旭川ヤマザキ集荷・クレート配達）と②（厚木バース差し替え）が実務では必ずセットで届くこと、かつ両方の情報が揃わないと出力1の件名（差し替え相手の車番）も出力2の切り場所（厚木/青梅）も正しく埋まらないことが判明。解決はコード変更ではなく「①②を1つの指示書としてまとめて入力する」という運用上の対応で足りた（YAGNI通り、まず本当に新しい仕組みが要るか確認してから作る、を実践）
- `input()`は改行（Enter）が来るまでの1行しか読み取らない。複数の指示書をまとめて渡したい時は、改行で区切ると2件目以降が次の`input()`ではなくその後のシェルに渡ってしまう（プログラム終了後の状態でシェルが余った文字列をコマンドとして解釈しようとしてエラーになる）。1行にまとめて渡す必要がある
- 逆に、②（差し替え・厚木バース切り）だけを単独でテストすると、①の「旭川ヤマザキ集荷」という前提情報が指示書内に無いため、AIが「テンプレート通りの集荷完了報告」と「実態に近い切り離し報告」の両方を勝手に生成し、本来2通りのはずが3セクションに増えるという混乱が発生した。これは実際の運用（①②は必ずセットで届く）では起きないケースであり、「単体でテストして良いケース」と「セットでしか意味をなさないケース」を区別する必要があると学んだ
- タイプミス（`[]切り離し場所]切り`→`[切り離し場所]切り`）を修正し、出力1側の未使用だった`【切り場所】`欄も「まだ確定していない情報は空欄で出力」というルールに合わせて空欄化。Day5で学んだ「個別列挙より範囲指定」の書き方（【行き先】から【切り場所】まで、を省略せず出力）をパターン3にも自力で適用し、common/branches化後も差し替えルール（集荷完了＝差し替え相手の車番／空車回送＝元の車番）が崩れていないことを実API確認した
- detection_promptの判定結果が稀に`pattern_id`と完全一致せず`StopIteration`で落ちる現象を発見。`repr()`でAIの生の返り値を可視化するデバッグ手法を学んだ（`print`だけだと空白や改行が見えず原因を誤認しやすい）。3回再現しなかったため、Day6のYAGNI方針（先回りせず壊れたら直す）に従い一旦保留とした
- 「同じことを繰り返さない」の原則から、全パターン共通のルールをJSON最上位に`all_common`として切り出す設計を検討した。実際に全パターンを比較したところ、一言一句完全一致する文言は「Markdownを使わずプレーンテキストで出力すること」の1つだけで、他の類似ルールはフィールド名や対象が異なり見た目が似ているだけと判明。「重複＝完全一致」で線引きする必要があることを学び、今回は抽象化コストに見合わないためYAGNIで実装を保留した
- 出力に紛れ込む見出し行（【出力1：出発報告】等）が本文と混同されるか検証。実際の出力では見出しと「件名：」行が必ず分離しており、かつこのツールは自動送信ではなく「人間が件名以降を目で見てコピーする」運用のため、実害がないと判断して対応不要と結論づけた。仕組みを直す前に「そもそも実際に困るのか」を運用ベースで確認する視点を実践した

## Day 8で習得したこと（2026/07/09〜10）
- 「①②」という手がかりは集荷回数ではなく、一日の複数車両・工程を区別する通し番号だった。実指示書を見せてもらって初めて前提が間違っていたと判明し、仮説は実物のデータで検証する必要があることを体感した
- `.count()`メソッド：文字列中に特定の部分文字列が何回出現するかを数える。「集荷後」という言葉の出現回数を数えることで、集荷回数Nをコード側で決定的に（AIに頼らず）取得できるようにした
- 「答えが決定的に決まる判断はコードに、自然文の抽出・生成はAIに」というDay5の原則を、集荷回数の可変対応という新しい課題にも再適用できることを確認した
- 文章を物理的に`.split()`で切り刻むと、切り離した断片から車番などの前提情報が失われてAIが埋められなくなる、という失敗パターンを検討。代わりに「元の指示書は毎回まるごと渡しつつ、コード側から『N回中i回目の集荷分だけ書いて』という一文を追加する」設計に至った
- 「小さく検証してから本組み込みする」というスパイク的な開発手順は初心者だからやる代替手段ではなく、実務でも使われる正当なやり方だと確認できた（CLAUDE.mdの「まず動くコードを書く→動いたら整える」の実践そのもの）
- 呼び出し回数が分岐によって異なる場合、「最後に共通で1回呼ぶ」という設計は成立しなくなる。各分岐の中でそれぞれ`generate_and_print()`を呼ぶ形に変更する必要があった
- 関数定義（`def`）のブロックは、呼び出しに必要な行**全部**を同じインデント幅に揃える必要がある。一部の行だけインデントしても`IndentationError`は消えるが、揃ってない行は関数の外に出てしまい、後続処理で想定外のタイミングで実行される（`message`が未定義のまま参照される等）
- 分岐の中で「本来は毎回呼ぶはずの処理」を`if not matched:`のようなネストしたブロックの中に書いてしまうと、その条件を通らないケース（`matched=True`の場合）ではその処理が一度も実行されない、というバグを作り込みかけた
- 上記バグの修正方法として「マッチした側にも呼び出しを書き足す（選択肢A）」と「共通の呼び出しを1箇所に集約する（選択肢B）」を比較。「同じコードが何箇所に存在するか」という基準でDRY原則を自力で適用し、選択肢B（`generate_and_print(prompt)`をif/elseの外に1回だけ出す）を選んだ。パターン3のクレート配達branchで①②結合入力の実API確認まで完了
- パターン1の実指示書テスト（集荷2回）で、現場目線のチェック項目を2つ発見：①出発報告2回目の【積み荷】欄が累積表示になっていない（その回に集荷した分だけしか載らない）、②出発報告1回目と2回目で【到着地点】欄の埋まり方が不揃い（片方は空欄、片方はAIが推測して埋めている）。原因は`departure_template`の「必ず空欄のまま出力すること」という指示リストに【到着地点】が含まれておらず、Day5で学んだ「省略されると困る空欄フィールドは個別に明示しないとAIが勝手に埋める」の再発だと特定
- 文字列の`.count("")`（空文字列を数える）は、その文字列の長さ＋1を返す。`repeat_keyword`キーを持たないパターン（3など）で`selected.get("repeat_keyword","")`が空文字列になり、`pickup_count`に意味のない大きな数字（74等）が出るデバッグ表示の謎を確認。実害は無いが、デバッグ表示としては紛らわしい
- few-shot（実例をプロンプトに埋め込む）と抽象ルールの使い分け：自然文からの情報抽出で答えが一意に決まらない場面（「①②」と「車番」の混同）では、実例を並べるfew-shotが有効な対策になり得る。ただし実例が1個だけだと、AIが意図した意味的なルールではなく「たまたまその実例にあった位置関係」を拾ってしまう「ショートカット学習」のリスクがあると学んだ（位置がバラバラな複数実例を混ぜるのがその対策）。ただし今回は実例を使わずとも、他パターン（3・4・5）に既にあった明示ルール「①②等の丸数字は無視」をパターン1に追加するだけで解決できた
- AIは「空欄にする」という指示より「読み取れる情報はとりあえず書きたい」という性質の方が強く出ることがある（出発報告の【切り離し場所】欄のケース）。広い範囲指定ルールの中に埋もれた項目は軽視されやすく、該当項目だけを名指しして「たとえ情報が読み取れても空欄にする」と否定形で個別に強調すると効いた
- JSON文字列は`\n`エスケープでしか改行を表現できないという構造的な限界により、長いテンプレートは1行の巨大な文字列になり読みにくくなる。解決策として、テンプレート本文を別の`.txt`ファイルに切り出し、`patterns.json`側はファイル名だけを持つ設計（コードとデータの分離をさらに一段進めた形）を採用。新しいライブラリやYAMLのような新しい文法を持ち込まずに、Day2で覚えた`open()`だけで解決できた
- プロの現場でも「プロンプト本文」と「設定・制御ロジック」を分離する設計は一般的で、YAML（ブロックスカラー`|`）を使うか単純な別ファイル参照にするかはチームの好み・規模次第という位置づけ。今の規模では新しい依存（pyyaml）を増やさない別ファイル参照で十分と判断した
- Claude APIの各呼び出しは完全にステートレスで、過去のやり取りを一切記憶しない。「AIに指示書を投げ続けて学習させる」ことはできず、ファインチューニング（モデルの重み自体を再学習、高コストでこの規模では不要）・few-shot（プロンプトに実例を埋め込む、今回採用）・RAG（似た実例を動的に検索して注入、Version3で検討）という3段階の選択肢があると整理した
- `while True:`：`for`文が「リストを使い切ったら自動で終わる」有限の繰り返しなのに対し、`while True:`は「回数が決まっていない、明示的にやめるまで続く」繰り返しという別の制御構造。プログラム全体をこれで包み、`if instruction == "終了": break`をinput直後（他の処理が走る前）に置くことで、`python mail_generator.py`を1回起動するだけで複数の指示書を連続処理できるようにした。ループの開始位置（inputより前）と終了判定の位置（inputの直後、他の処理より前）を間違えると、それぞれ「毎回同じ指示書で回り続ける」「終了時にパターン判定ロジックが誤動作してエラーになる」という別々の不具合を生むことも実際に検討して理解した
- パターン3のkeywordsに含まれていた「℃」が原因で、ヤマザキ・旭川と無関係な指示書（実際はパターン1相当）が誤ってパターン3と判定される実バグを発見。「℃」は温度が書かれたほぼ全ての指示書に出現するため、パターンを見分ける情報として機能していなかった。Day4で学んだ「keywordsの衝突は本当にそのパターンが独立して必要か疑うきっかけ」と同じ構造で、今回は独立性ではなく「汎用的すぎる語を安易にkeywordsへ入れない」という追加の教訓を得た
- 実際に届く業務メール（同じシャーシ番号で1日に複数の配達・集荷が連続するケース）を丸ごとテストしたところ、「1回のペーストから複数指示書を自動分割する」機能を作る前に、既存パターンの設計自体に2つの穴があることが判明。**課題A**：パターン4は「配達完了メール」の後に必ずシャーシ切りか空車回送のどちらかを続ける設計だが、実際には「配達だけで完結し、後に何も続かない」指示書が存在し、今の設計では情報が無いのにAIが後続メールをでっち上げるリスクがある。**課題B**：「集荷して、そのまま別の場所へ配達して完結する」（集荷完了メール＋配達完了メールの2通、切りも空車回送も無い）指示書は、パターン1（集荷して切りで終わる）ともパターン4（配達して切り/空車回送で終わる）とも異なる3つ目の型で、既存のどのパターンにも属さないと判明
- 「AIに指示書を丸ごと渡して自動で全部判断させれば早いのでは」という問いに対し、これまでの実装で繰り返し体験してきた「答えが決定的に決まる判断はコードに、自然文の理解・生成だけAIに」という原則を、スケール（複数パターン・複数指示書の一括処理）の議論にも再適用して整理した。丸投げは開発コストを下げる代わりに検証コストが跳ね上がり、結局「AIの出力を毎回全部人力チェックする」羽目になるため、実務レベルのツールでは成立しないと結論づけた
- CLAUDE.mdが会話開始時に毎回全文読み込まれる仕組みだと理解し、日々の詳細な学習記録を追記し続けると「今すぐ必要な情報」が「決着済みの歴史」に埋もれていくと気づいた。「現在の状態のスナップショット（CLAUDE.md）」と「積み上げ型の歴史ログ（LEARNING_LOG.md、このファイル）」を分離するという、Claude自身の記憶システムと同じ設計原則を適用した

## Day 10で習得したこと（2026/07/23）
- Day9で受け取った参考実指示書を実際に読み込み、前提を1つ修正した。「全工程が同じ車番で繋がっている」「①②③が指示書をまたぐ通し番号」という理解は、実際には「空行区切りの1ブロック＝独立した1通の指示書」であり、Finが参考用に複数の別々の指示書をまとめて貼っていただけだったと判明。1通の指示書内では①②③はそのつど1から始まり、レグごとに車番が異なることもある。「残りのちほど」「指示書苫小牧確認」等のメモ書きはFin自身が判断する情報であり、ツールに渡す指示書テキストには含まれないと確認した
- 1通の指示書内で①②③が複数ある場合の分割ロジックを、Claudeとの問答形式で設計した。核心は「`split()`は1回の呼び出しで対象文字の全出現箇所を目印にできる」という性質（`"りんご、みかん、ぶどう".split("、")`が1回の呼び出しで3分割される例で確認）。①②③という3種類の異なる目印を1回の`split()`で扱うには、事前に`.replace()`で②③を①に統一する必要があると自力で導出できた
- 処理順序をA（②→①に置換）→B（③→①に置換）→C（①でsplit）→D（レグごとに既存のパターン判定・生成処理をループで回す）と確定。C を A・B より先にやると②③がまだ①になっておらず正しく分割できないことを、具体例のトレースで確認した。split後の先頭に空文字列が混入する点（文字列の先頭が目印そのものから始まるため）も具体例で把握した
- 実装は、分割後の空文字を除いて`legs`リストに追加する`.append()`を書く1行手前で中断。ここまでの設計・進捗をCLAUDE.md/LEARNING_LOG.mdに記録し、次回はコードの実装から再開する
